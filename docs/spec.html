ooze — Repository Specification

Multi-language mutation-testing CLI. It scans a repository, generates semantic source mutations, runs your existing test suite against each one in an isolated workspace, and reports which mutations your tests failed to catch (the survivors). Surviving mutants flag code paths your tests don't truly exercise.

Rust • edition 2024 v0.1.0 CLI binary: ooze 23 languages tree-sitter
How this document was produced

This spec is derived strictly from the repository itself — README.md, docs/, Cargo.toml, ooze.example.toml, and the src/ tree (notably main.rs, core.rs, scheduler.rs, runner/, report/, config/, skip/, doctor.rs). Statements that could not be confirmed from the repository are explicitly marked Unknown from current context or Needs confirmation.

01Project purpose

ooze exists to measure test-suite effectiveness beyond line coverage. Line coverage tells you a line ran; mutation testing tells you whether a test would fail if that line were subtly wrong. ooze introduces small, behavior-changing edits ("mutants") into source code and re-runs the project's own test command (the "probe") against each one:

Killed

A test failed — the suite caught the mutation. Good.

Survived

Tests still passed — a gap in the suite. Actionable signal.

Timeout

Probe exceeded the per-mutant wall-clock cap.

Error

The probe could not run / infrastructure failure.

It is explicitly designed to be CI-friendly: deterministic exit codes, multiple machine-readable report formats (JSON, SARIF, GitHub annotations, agent task lists), and a --changed-only mode for fast PR runs.

02Scope of this repository

This repository is the entire ooze tool: a single Rust crate producing one binary. Its responsibilities:

  • Discover functions across 23 languages via tree-sitter grammars (src/lang/, queries/).
  • Compute cyclomatic complexity and CRAP scores, optionally fused with LCOV coverage (src/crap/).
  • Generate semantic mutation candidates from per-language operator implementations (src/mutate/).
  • Statically skip equivalent / non-meaningful candidates (src/skip/).
  • Order candidates by strategy and optional coverage signal (src/scheduler.rs).
  • Materialize isolated per-worker workspaces (copy or overlayfs) and run the probe in parallel (src/runner/).
  • Classify outcomes and emit reports in several formats (src/report/).
  • Load and merge configuration from ooze.toml + CLI flags (src/config/).
  • Diagnose preconditions (src/doctor.rs).

03What this repository does not cover

  • It does not run or manage your test framework. You supply the probe command after --; ooze only executes it and reads its exit status.
  • It does not collect coverage. LCOV is consumed as input (--lcov); generating it is out of scope.
  • It is not a linter or formatter. Mutations are throwaway, applied only inside disposable workspaces — source on disk is never modified by a run.
  • It ships no GitHub/GitLab pipeline of its own — docs/ci.md provides recommended snippets, but no .github/ workflow exists in the repo (see §27).
  • Languages on the backlog are not yet mutated — see docs/deferred-languages.md.

04System overview

ooze is a stateless command-line program. Each invocation reads source files (and optionally a config and an LCOV file), runs a pipeline, and writes a report to stdout or a file. The only persistent state on disk is the cache and run scratch directories under .ooze/.

05Core concepts and glossary

TermMeaning (as used in this repo)
Mutant / candidateA single semantic edit at a known byte range (MutationCandidate in core.rs). Carries id, file, language, operator, original/replacement text.
OperatorA language-agnostic mutation idea (OperatorName), e.g. negate_equality. Nine exist today.
ImplementationOne language's realization of an operator (MutatorImpl), id like rust.negate_equality, defined in src/mutate/registry.rs.
ProbeThe user-supplied test command (everything after --, or [probe].command). Its exit status decides killed vs survived.
PreflightOne probe run on unmodified code to confirm the baseline is green before mutating.
WarmupPre-building the probe in each worker's build cache so the first mutant isn't a cold compile.
WorkspaceAn isolated copy (or overlayfs mount) of the repo where one mutant is applied and probed.
CRAPChange Risk Anti-Patterns score: combines cyclomatic complexity and coverage to flag risky functions (src/crap/).
StrategyCandidate ordering: discovery, actionable, highest_crap (scheduler.rs).
Verdict / statuskilled, survived, timeout, error (MutantStatus).
Static skipA rule that drops candidates unlikely to be meaningful (test files, generated files, assert/panic macro internals).

09Architecture overview

The crate is organized as a pipeline of mostly-pure stages wired together in main.rs. core.rs holds the shared vocabulary; every other module depends on it and on little else.

Key architectural decisions (observed in code)

  • Operators are decoupled from grammars. lang/ only parses/discovers; mutation logic lives in mutate/registry.rs keyed by Language (per the doc-comment in lang/mod.rs).
  • Semantic operator vs per-language implementation split: one OperatorName can have many MutatorImpl rows; filters work at the semantic level so they apply across all languages.
  • Two workspace backends: full copy (portable) and overlay (overlayfs, Linux + root, no full copy per mutant), with auto selecting overlay when available.
  • Parallelism via rayon; each worker gets its own build cache (--per-worker-cache) to avoid build-lock contention.
  • Deterministic exit-code contract centralized in report::OozeExitCode.
  • Strict config: deny_unknown_fields — typos fail loudly. CLI flags always override config.
  • Tree-sitter byte ranges are the unit of edit; apply_mutation verifies the original text still matches before mutating, guarding against stale candidates.

11Development workflow

The repo uses a Nix flake dev shell (flake.nix) wired through direnv (.envrc). It provisions a stable Rust toolchain (with rust-src, rust-analyzer, llvm-tools-preview), jq, and llvm-cov/llvm-profdata for coverage.

# enter dev shell (automatic with direnv; otherwise)
nix develop

# build + test
cargo build --release          # binary: ./target/release/ooze
cargo test                     # unit tests + tests/cli.rs integration tests
cargo clippy                   # pedantic lints (warn) per Cargo.toml

# dogfood ooze on itself
./target/release/ooze test-mutants --path . --strategy actionable \
  --per-worker-cache --warmup --probe-env CARGO_TARGET_DIR={build_cache} \
  -- cargo test
Convention

ooze is dogfooded on itself: lcov.info is checked in and the Rust probe recipe is the canonical example throughout the README and docs/. New mutation operators/grammars should be exercised against tests/fixtures/lang/sample.<ext>.

12Planning workflow

Unknown from current context

No formal planning artifacts (plan files, ADRs, RFC templates) exist in the repo. Git history is the only record of intent. The guidance below is the de-facto pattern inferred from structure and docs, not a documented process — Needs confirmation.

In practice, planning a change in this codebase means deciding which pipeline stage it touches and how it threads through the shared types:

  1. Locate the stage (grammar, operator, scheduler, runner, report, config).
  2. Check the shared contract in core.rs — does the change add an OperatorName, a Language, a report field?
  3. Use plan-mutants to preview selection/scoring effects without running probes — this is the built-in "dry run" for behavior changes.
  4. Update docs/ in lockstep (config.md mirrors schema.rs; README mirrors the CLI).

The plan-mutants command is itself the closest thing to a "planning workflow" the tool offers to its users: it shows total candidates, skipped count, applied excludes, the operator filter, and per-candidate selection scores and reasons — before any probe runs.

13Task execution workflow

A concrete code task typically follows the stage it lives in. Two common ones:

Adding a mutation operator (one language)

  1. Add an OperatorName variant + OperatorInfo (category, default_enabled, description, test_hint) in core.rs; register in OperatorName::ALL.
  2. Add a tree-sitter query under queries/<lang>/ (see existing Rust operator queries).
  3. Add a MutatorImpl row (query + replacement + description) in src/mutate/registry.rs, id <lang>.<operator>.
  4. Add a fixture case in tests/fixtures/lang/sample.<ext> and assert via mutants/plan-mutants.

Adding a language for discovery

  1. Add the tree-sitter-<lang> dependency in Cargo.toml.
  2. Add a Language variant in core.rs (with as_str/parse).
  3. Add src/lang/<lang>.rs with a GrammarDef and queries/<lang>/functions.scm + branches.scm.
  4. Register the module in src/lang/mod.rs and add a fixture sample.

For mutation runs, always pass --preflight in validation: it runs the probe on unmodified code first and aborts with exit code 2 if the baseline is broken, preventing meaningless "all survived" reports.

15Human review checkpoints

  • New operator semantics — does the mutation produce a genuinely different behavior (not an equivalent mutant)? Review the replacement fn and skip rules.
  • Exit-code / report-format changes — these are an external contract for CI; verify against docs/ci.md.
  • Config schema changes — deny_unknown_fields means additions must update docs/config.md and ooze.example.toml together.
  • Workspace/runner changes — isolation and cleanup correctness; overlayfs requires root and is Linux-only.
  • Default-on operators — flipping default_enabled changes everyone's run; needs explicit sign-off.

16Agent working instructions

Convention for agents

Before editing, read the relevant module's leading doc-comment — they encode non-obvious decisions (why operators left lang/, why Language keeps a legacy serialized string, why specific clippy lints are allowed). Honor those rather than "fixing" them. Never modify source outside the intended stage to silence a clippy warning; the allowed exceptions are deliberate and centralized in Cargo.toml.

When the task is "address survivors" rather than "change ooze," run ooze test-mutants --format agent-tasks-json (or markdown) and use each task's file/line/operator/test-hint to write a targeted test — do not weaken the mutant or the probe.

17Team working instructions

  • Develop inside the Nix dev shell so toolchain and coverage tools match.
  • Keep README.md, docs/config.md, and ooze.example.toml consistent with src/config/schema.rs — they are tested-by-reading, not by CI.
  • Add a fixture for every new operator/language; integration tests in tests/cli.rs drive the real binary.
  • Treat exit codes and report formats as a public contract; bump behavior deliberately.
  • Prefer extending the registry over special-casing in main.rs.

23Error handling and failure modes

CodeMeaning
0Completed; no survivors, no timeouts/errors.
1Completed; survived mutants found.
2Preflight failed/timed out — probe broken on clean code.
3Completed but timeout/error outcomes occurred (incomplete).
4Usage / invalid invocation (reserved).
5Internal ooze error (reserved).

Priority when several apply: preflight > infrastructure > survivors > success. Overrides: --no-fail-on-survivors (exit 0 despite survivors) and --allow-incomplete (timeouts/errors non-fatal).

Other failure modes handled in code: invalid/mismatched candidate byte ranges abort the apply (apply_mutation verifies original text), unknown config keys and unknown operators/categories fail at load, and --changed-only errors clearly if --path isn't inside a git repo. Recoverable internals use anyhow with context; typed errors use thiserror.

24Security and privacy considerations

  • Arbitrary command execution by design: the probe is whatever you pass after --. ooze runs it repeatedly — only run trusted probes on trusted repos.
  • overlayfs backend needs root. Prefer the copy backend in untrusted or unprivileged environments.
  • No network calls are made by ooze itself; it reads local files and spawns the probe/git.
  • Reports may contain source snippets and probe stdout/stderr. Trim with --report-detail compact / --no-stdout / --no-stderr before publishing to avoid leaking secrets that tests print.
  • Workspaces are isolated copies/overlays, so a misbehaving mutant cannot corrupt the real working tree.

25Observability and diagnostics

  • Progress on stderr (--progress auto|always|never, --quiet): per-mutant [i/N] STATUS id lines.
  • Structured reports (JSON/SARIF) carry per-outcome diffs, durations, exit codes, and optional probe output.
  • ooze doctor emits human or JSON check results (Ok/Warn/Fail/Skip) for preconditions.
  • plan-mutants exposes selection scores and reasons for transparency into why a candidate was (de)prioritized.
  • tracing / tracing-subscriber are dependencies for instrumentation. Extent of tracing usage: Unknown from current context

27Known constraints

  • Rust edition 2024 toolchain required to build.
  • overlay backend: Linux + overlayfs + root only; elsewhere falls back to copy.
  • --changed-only requires the path to be inside a git repository.
  • Mutation is limited to languages with an operator implementation; discovery covers all 23 grammars but mutation operators are richest for Rust.
  • Per-mutant cost ≈ one full probe run; wall time scales with candidate count ÷ jobs. --limit and --changed-only bound it.
  • Reports can be large (diffs + probe output per outcome) — trimming flags exist for this reason.

28Known risks

  • Equivalent mutants: some surviving mutants are behaviorally identical and unkillable; static skips mitigate but cannot eliminate them — survivor counts need human judgment.
  • Flaky probes: a non-deterministic test suite produces noisy killed/survived results; --preflight and a sane --timeout-seconds reduce but don't remove this.
  • Shared build cache contention if --per-worker-cache is omitted under high --jobs.
  • Documentation drift: docs/ mirrors code by convention, not by test — easy to desync.
  • No CI workflow in-tree means quality gates depend on discipline / an external pipeline.

30Anti-patterns

  • ❌ Running without --preflight on CI — a broken baseline yields all-survived noise.
  • ❌ "Fixing" a survivor by deleting/weakening the mutant or loosening the probe instead of adding a real test.
  • ❌ Adding config keys without updating docs/config.md + ooze.example.toml (and remembering deny_unknown_fields).
  • ❌ Special-casing a language in main.rs instead of extending the registry/grammar layer.
  • ❌ Silencing clippy with scattered #[allow] when the project centralizes intentional exceptions in Cargo.toml.
  • ❌ Using the overlay backend in unprivileged/untrusted contexts.
  • ❌ Publishing full reports with probe output that may contain secrets.

32Final summary

ooze is a focused, single-crate Rust CLI for multi-language mutation testing. Its architecture is a clean pipeline — discover (tree-sitter) → mutate (operator registry) → skip → schedule → run in isolated parallel workspaces → classify → report — anchored by shared types in core.rs. It is built for CI with deterministic exit codes, several report formats (including SARIF and agent task lists), and a --changed-only fast path. The project documents itself through docs/ and the README rather than an SCE context/ directory, and it dogfoods on its own test suite. The main areas a newcomer should verify are the CI story and the release plan, both of which are not pinned down in the tree today.
