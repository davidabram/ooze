Onboarding ooze into an existing project

A practical strategy for introducing mutation testing into a brown-field codebase without making the team resent it. Start ooze as a test-improvement advisor scoped to changed code, generate concrete test-writing tasks, and tighten into a gate only once the signal is trusted.

brown-field rollout advisory → gate changed-only scope 9 phases
How this document was produced

Every command, flag, and config key below was verified against the ooze source in this repository — src/main.rs, src/core.rs, src/scheduler.rs, src/config/schema.rs, and docs/config.md. Run ooze --help, ooze <command> --help, and ooze operators to confirm the surface for your build. A Markdown version lives at docs/onboarding.md.

00Guiding principle

Do not start by mutating the whole repo. For brown-field code, the goal of the first few weeks is not a coverage gate — it is to surface a small number of high-value survived mutants and turn them into concrete test-writing tasks. Start ooze as a test-improvement advisor, scoped to code the team is already changing, in advisory (non-gating) mode. Tighten only once the signal is trusted.

Killed

A test failed — the suite caught the mutation. Good.

Survived

Tests still passed — a gap in the suite. This is the actionable signal.

Timeout / Error

Infrastructure problems. Keep these visible and separate from survivors.

01What ooze actually mutates

ooze ships a fixed set of small, high-signal operators — boolean, comparison, equality, None/null, and empty-collection mutations. List the ones built into your binary with:

ooze operators
Myth to drop

There are no arithmetic, string, regex, or statement-removal operators in ooze, so there is nothing of that kind to "turn off." The only default-off operator is integer_zero_one — opt into it explicitly via the operators allow-list if you want it. An operator only runs for languages that implement it; see docs/config.md for the per-language breakdown.

02Phase 1 — Baseline

Goal: prove the probe is stable scope: local gating: none

Confirm the existing test command is stable and ooze's preconditions pass.

ooze doctor --path .

Generate a starter config:

ooze init-config --language rust

Edit ooze.toml so the probe runs your normal test command:

[probe]
command = ["cargo", "test", "--jobs", "1"]

[runner]
preflight = true
timeout_seconds = 120
jobs = 1
workspace_backend = "auto"
cache_dir = ".ooze/cache"
runs_dir = ".ooze/runs"

[report]
format = "human"
fail_on_survivors = false
allow_incomplete = false

Smoke-test a single mutant:

ooze test-mutants --limit 1

Expected: preflight passes, one mutant runs, the original tree is left unchanged. Fix any runner/probe/config issues before widening scope.

test-mutants vs test-mutant

test-mutants (plural) runs a batch and produces a report. test-mutant (singular) runs exactly one mutant by id — handy for debugging, not for normal runs.

03Phase 2 — Changed code only

Goal: focus on code the team is editing scope: diff vs base gating: none

For brown-field repos, scope to what the team is editing:

[scope]
changed_only = "main"

or per-invocation:

ooze plan-mutants --changed-only main --limit 20

plan-mutants shows the selection, scores, and applied excludes without running any probes — use it to sanity-check scope before spending test time. A good first real run:

ooze test-mutants \
  --changed-only main \
  --strategy actionable \
  --limit 10 \
  --format agent-tasks-markdown \
  --output ooze-tasks.md \
  --no-fail-on-survivors

04Phase 3 — Exclude bad mutation targets

Goal: mutate business logic, not codegen

Keep ooze on business logic, not generated code, fixtures, or snapshots. DEFAULT_EXCLUDES (.git, target, .ooze, node_modules, vendor, __pycache__, .gradle) and .gitignore are always applied; add the rest:

[scope]
exclude = [
  "dist/**",
  "build/**",
  "coverage/**",
  "**/generated/**",
  "**/*.generated.*",
  "**/*_pb2.py",
  "**/*.pb.go",
  "**/fixtures/**",
  "**/testdata/**",
  "**/__snapshots__/**",
  "**/snapshots/**",
]

05Phase 4 — Use coverage to rank, not to punish

Goal: prioritize, don't gate

Feed coverage in so ooze can prioritize; don't turn it into a gate yet. Each --coverage entry is format:path (lcov, cobertura, jacoco, go-cover) or a bare path to auto-detect; multiple entries merge.

ooze test-mutants --coverage lcov:coverage/lcov.info
ooze test-mutants --coverage cobertura:coverage.xml
ooze test-mutants --coverage go-cover:coverage.out
ooze test-mutants --coverage jacoco:build/reports/jacoco/test/jacocoTestReport.xml

With the actionable strategy, the sweet-spot targets are functions with moderate CRAP, some existing coverage, small enough to understand, and fast tests.

Good first targets

Moderate CRAP, partial coverage, small business-logic functions, fast tests.

Avoid at first

CRAP-100 god-functions, 0%-coverage code, framework wiring, slow database/network-heavy integration flows.

06Phase 5 — Generate agent tasks

Goal: turn survivors into work

The strongest brown-field workflow turns survivors into specific test-writing tasks:

ooze test-mutants \
  --changed-only main \
  --strategy actionable \
  --limit 25 \
  --format agent-tasks-markdown \
  --output ooze-tasks.md \
  --no-fail-on-survivors

Hand ooze-tasks.md to a developer or an AI agent. Each task names the file, function, line, the surviving mutation, source context, and CRAP/coverage — i.e. exactly what's needed to write a targeted test.

07Phase 6 — CI in advisory mode

Goal: PR feedback without blocking gating: none

Start CI non-blocking with GitHub annotations:

name: ooze

on:
  pull_request:

jobs:
  mutation-advice:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build ooze
        run: cargo build --release

      - name: Run ooze
        run: |
          ./target/release/ooze test-mutants \
            --changed-only origin/main \
            --strategy actionable \
            --limit 25 \
            --jobs 2 \
            --timeout-seconds 120 \
            --format github-annotations \
            --no-fail-on-survivors \
            -- cargo test --jobs 1

Prefer SARIF? Swap the format and upload it:

      - name: Run ooze (SARIF)
        run: |
          ./target/release/ooze test-mutants \
            --changed-only origin/main \
            --strategy actionable \
            --limit 25 \
            --jobs 2 \
            --timeout-seconds 120 \
            --format sarif \
            --output ooze.sarif \
            --no-fail-on-survivors \
            -- cargo test --jobs 1

      - name: Upload SARIF
        uses: github/codeql-action/upload-sarif@v3
        with:
          sarif_file: ooze.sarif

Everything after -- is the probe command. See docs/ci.md for more.

08Phase 7 — Soft team policy

Goal: a norm, not a gate

Once the output is trusted, agree on a rule (still without failing CI):

Team rule

If ooze finds a survived mutant in changed business logic, either add a test, mark it intentionally ignored, or explain why the survivor is acceptable.

Keep --no-fail-on-survivors on and require humans to review the tasks.

09Phase 8 — Gate changed code only

Goal: new work kills its mutants gating: changed code

When the team is ready, drop --no-fail-on-survivors for changed code so new work must kill its mutants:

ooze test-mutants \
  --changed-only origin/main \
  --strategy actionable \
  --limit 50 \
  --format human

Keep allow_incomplete = false so infrastructure failures (timeouts/errors) still surface separately from survivors. Do not gate the historical repo yet.

10Phase 9 — Widen scope in slices

Goal: expand without a cliff

Expand module by module, not all at once:

ooze test-mutants --path crates/domain --strategy actionable --limit 50
# later
ooze test-mutants --path crates/domain --strategy actionable --limit 100

11Reference config for brown-field repos

This parses against the current schema. The config uses deny_unknown_fields, so a typo or an unsupported key (for example shared_target) is rejected as a load error rather than ignored — only use keys listed in docs/config.md.

[scope]
changed_only = "main"
exclude = [
  "dist/**",
  "build/**",
  "coverage/**",
  "**/generated/**",
  "**/*.generated.*",
  "**/fixtures/**",
  "**/testdata/**",
  "**/__snapshots__/**",
  "**/snapshots/**",
]

[mutation]
strategy = "actionable"
limit = 25
static_skips = true
context_lines = 3

[runner]
workspace_backend = "auto"
jobs = 2
timeout_seconds = 120
preflight = true
warmup = true
per_worker_cache = true
cache_dir = ".ooze/cache"
runs_dir = ".ooze/runs"

[probe]
command = ["cargo", "test", "--jobs", "1"]
env = ["RUST_BACKTRACE=1"]

[report]
format = "agent-tasks-markdown"
output = "ooze-tasks.md"
detail = "compact"
fail_on_survivors = false
allow_incomplete = false
diff = false
stdout = false
stderr = true
only_survivors = true
On the operators key

The [mutation] operators / exclude_operators keys are optional. By default every operator implemented for your language runs except integer_zero_one. Only add an operators allow-list if you have a reason to narrow it — there are no noisy arithmetic/string/regex operators to exclude.

12Tracking adoption

Good signs

Survivors lead to real tests; timeouts/errors are rare; the same functions stop reappearing; developers understand the generated tasks.

Bad signs

Lots of equivalent/noisy mutants; frequent timeouts; mutating fixtures or generated code; oversized reports; ignored annotations.
